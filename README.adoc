= elx — Enterprise Linux Base Image
:toc: macro
:toclevels: 3

_A Packer-built, pre-hardened, batteries-included Enterprise Linux base image for building virtual machines._

image::screenshot.png[Screenshot with some glimpses of the workflow.]

`ELx` is a heavily-modified Enterprise Linux image that is pressed using https://www.packer.io/[Packer]. This repository is responsible for applying identical bootstrapping tasks to multiple releases of Enterprise Linux, e.g. Rocky Linux to automates the creation of a
customized, minimal, ready-to-use Enterprise Linux (RHEL / Rocky Linux 9) virtual-machine image that you can boot up and use for cli-based workflows such as system administration or as a daily-driver for Linux professionals that is configured visually and operationally uniformly as new releases of EL come out.

In other words, instead of installing an OS by hand and then spending an afternoon configuring shells, editors, login banners, SSH hardening, and installing your favorite command-line tools, you run a single `packer build` and get back a reproducible VM image with all of that baked in and shipped with the same familiar custom `ELx` window manager, terminal, and cli tool suite.

toc::[]


== What's inside the image

=== Base packages (installed during OS install via Kickstart)

Installed as RPMs by the Kickstart `%packages` section, e.g. `aide`, `bash-completion`, `vim-enhanced`,
`tmux`, `zsh`, `perl`, `bind-utils`, `openssh-clients`, `sudo`, `rsync`, `wget`, `net-tools`, `nfs-utils`,
and other common administration utilities.

=== Shell & editor experience

[cols="1,3"]
|===
| Component | Configuration applied

| *zsh*  | Set up with the https://github.com/sindresorhus/pure[Pure] prompt; system-wide config in `/etc/zshrc`.
| *tmux* | Auto-attaches a session on login and detaches/logs out cleanly (toggle with `TMUX_AUTO_ATTACH`). Requires tmux ≥ 3.2.
| *vim*  | Preloaded with the https://github.com/sheerun/vim-polyglot[vim-polyglot] language pack (system `vimfiles`).
|===

| https://github.com/sharkdp/fd[fd]        | Fast, user-friendly `find`            | 8.7.0

NOTE: Most versions are pinned inside each `template/scripts/install_*.sh` script — bump them there to
upgrade a tool. `rg` and `htop` are installed from EPEL and track whatever version EPEL ships for the
target Enterprise Linux release, so they work on both EL9 and EL10.

=== System hardening & housekeeping

* *SSH* — `UseDNS no` and `PasswordAuthentication no` set in `sshd_config`.
* *Login banners* — authorized-use warnings written to `/etc/issue` and `/etc/issue.net` (`issue.sh`),
  and a branded MOTD (`motd.sh`).
* *OS metadata* — `VARIANT`, `BUILD_ID`, and `DOCUMENTATION_URL` stamped into `/etc/os-release` so the
  image is self-describing (`metadata.sh`).
* *Networking* — predictable `eth0` naming and a `single-request-reopen` DNS latency fix.
* *SELinux* is left in `permissive` mode; the firewall is disabled (see the Kickstart file to change this).

== How it works

[source]
----
                    ┌──────────────────────────────────────────────┐
   packer build     │  1. parallels-iso builder boots the EL9 ISO  │
        │           │  2. Serves http/el9/ks.cfg over HTTP          │
        └──────────▶│  3. Kickstart does an unattended install     │
                    │  4. Reboots, Packer connects over SSH         │
                    │  5. shell provisioners run scripts/*.sh       │
                    │  6. cleanup.sh + minimize.sh shrink the image │
                    │  7. Image exported to builds/                 │
                    └──────────────────────────────────────────────┘
----

* The build is defined in `template/template.pkr.hcl`, with all of the tunable knobs (ISO, version,
  architecture, Kickstart file, credentials) declared in `template/variables.pkr.hcl`. It uses the
  *Parallels* builder (`parallels-iso`) and defaults to *Rocky Linux 10* on *aarch64 (Apple Silicon)*.
* Ready-made var-files select a target: `rocky10.pkrvars.hcl` (default) and `rocky9.pkrvars.hcl`.
* During install, Packer stands up a small HTTP server so the installer can fetch the Kickstart file
  named by the `kickstart` variable — `template/http/el10/ks.cfg` for Rocky 10, or `el9/ks.cfg` /
  `el8/ks.cfg` for older releases.
* The Kickstart file creates an unprivileged `ansible` build user with passwordless `sudo`. After first
  boot Packer logs in as that user over SSH and runs the provisioner scripts *in the exact order* listed
  in the `build` block, each via `sudo`.
* The last two scripts, `cleanup.sh` and `minimize.sh`, strip development packages and zero out free space
  so the exported disk compresses well.

== Requirements

* *Packer* — a recent release, in your `PATH`. Download: https://www.packer.io/downloads
* The *Parallels Packer plugin* (`github.com/hashicorp/parallels`, ≥ 1.0.3). `packer init` will fetch it.
* *Parallels Desktop* (Pro/Business) installed on the host — the default builder targets Parallels on macOS.
* An *Enterprise Linux ISO*. By default you don't need to download one manually: the build fetches the
  current *Rocky Linux 10* DVD straight from the Rocky mirror (see "Pointing at your ISO" to use a local
  or different image).

== Building

[source,console]
----
# 1. Fetch the required Packer plugin (first time only)
cd template
packer init .

# 2. (Recommended) validate the template
packer validate -var-file=rocky10.pkrvars.hcl .

# 3. Build the image
packer build -var-file=rocky10.pkrvars.hcl .     # Rocky Linux 10 (default)
# or
packer build -var-file=rocky9.pkrvars.hcl .      # Rocky Linux 9
----

`packer build .` with no var-file also works and produces the same Rocky 10 image, since the defaults in
`variables.pkr.hcl` already target Rocky 10. The finished image is written to
`builds/<os><version>_<arch>/` (e.g. `builds/rocky10_aarch64/`).

=== Pointing at your ISO

The ISO is chosen by two variables, set in the var-files (or overridable with `-var`):

[source,hcl]
----
iso_url      = "https://download.rockylinux.org/pub/rocky/10/isos/aarch64/Rocky-10-latest-aarch64-dvd.iso"
iso_checksum = "file:https://download.rockylinux.org/pub/rocky/10/isos/aarch64/CHECKSUM"
----

* `iso_url` may be an `https://` URL (downloaded and cached by Packer) or a `file:` path to a local ISO —
  drop ISOs into the `isos/` directory to keep them with the project.
* Using `iso_checksum = "file:<url-to-CHECKSUM>"` lets Packer read Rocky's published checksum file and
  match it to the ISO automatically, so builds keep working across point releases without editing a hash.
* The DVD image is used (not minimal) so the Kickstart `cdrom` install source has AppStream available.

To build against a different release, override on the command line, e.g.:

[source,console]
----
packer build -var 'os_version=10' -var 'iso_url=file:/Volumes/iso/my.iso' \
             -var 'iso_checksum=sha256:<hash>' -var 'kickstart=el10/ks.cfg' .
----

=== Building for x86_64 or another hypervisor

The template is configured for aarch64 + Parallels, but the tooling is largely architecture-aware:

* Set `-var 'arch=x86_64'` and `-var 'parallels_tools_flavor=lin'`, and point `iso_url` at an x86_64 ISO
  (e.g. `.../isos/x86_64/Rocky-10-latest-x86_64-dvd.iso`).
* Note that Parallels on Apple Silicon cannot run x86_64 guests — an x86_64 build needs an x86_64 host or
  a different hypervisor. To target QEMU, VirtualBox, or VMware, swap the `parallels-iso` source in
  `template.pkr.hcl` for the corresponding Packer builder and adjust the `boot_command`.

== Default credentials & security notes

WARNING: This image is intended as a *base for further automation*, not as a production-ready hardened
appliance. Review and change these before using it anywhere untrusted.

* The build user is `ansible` / `ansible` (used by Packer over SSH).
* The `root` password is set (encrypted) in the Kickstart file, and two SSH public keys are authorized for
  `root`. *Replace `rootpw` and the `sshkey` lines in `ks.cfg` with your own* before building.
* `PasswordAuthentication` is disabled — you will need one of the authorized keys (or console access) to
  log in to the built image.
* SELinux ships in `permissive` mode and the firewall is disabled; tighten these in the Kickstart file if
  your environment requires it.

== Iterating on the provisioner scripts (Docker)

A full VM build is slow to iterate on. To test/develop the install and configuration scripts quickly,
use the provided container, which drops you into a shell in the scripts directory on a fresh Rocky Linux 9:

[source,console]
----
docker build -t elx-scripts .
docker run --rm -it elx-scripts   # clones the repo and lands you in template/scripts
----

From there you can run scripts individually. `template/scripts/_test.sh` documents a convenient run order
for manual testing.

== Generating a single post-install script

If you want to apply the same provisioning outside of Packer (for example, against an existing host or a
cloud-init `runcmd`), `bin/gen_postinstall_script.sh` concatenates all the provisioner scripts — in build
order, stripping the redundant shebang lines — into one script printed to stdout:

[source,console]
----
cd bin
./gen_postinstall_script.sh > /tmp/postinstall.sh
----

== Customizing

* *Add a tool* — drop a new `install_yourtool.sh` in `template/scripts/`, then add it to the `scripts`
  list in the `provisioner "shell"` block of `template/template.pkr.hcl` (and, if you use them, to
  `_test.sh` / `gen_postinstall_script.sh`).
* *Change base packages* — edit the `%packages` section of `template/http/el9/ks.cfg`.
* *Change branding / banners* — edit `motd.sh` and `issue.sh`.
* *Change partitioning, timezone, locale, network* — edit the top of `ks.cfg`.
